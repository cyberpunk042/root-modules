# M007 — Connect Second Brain to root-modules

## Summary

Run `python3 -m tools.setup --connect-project $HOME` from inside the second brain. This single command (the canonical mechanism) wires root-modules as a sister project of the second brain by writing four artefacts INTO $HOME: a `research-wiki` MCP server entry in $HOME/.mcp.json, a `tools/gateway.py` forwarder, a `tools/view.py` forwarder, and a `## Second Brain Connection` block appended to $HOME/AGENTS.md. M007 is the load-bearing step of Stream 1 — it's what makes the second brain queryable from inside $HOME. M006 is the prerequisite (clean state, AGENTS.md exists, operator authorizes); M008 is the verification (smoke test from inside).

## Done When

- [ ] `python3 -m tools.setup --connect-project $HOME` runs from /opt/devops-solutions-information-hub and exits 0
- [ ] $HOME/.mcp.json contains a `research-wiki` server entry pointing at the second brain's venv-python + correct cwd
- [ ] $HOME/tools/gateway.py exists, executable, and `python3 -m tools.gateway --help` runs from inside $HOME and dispatches to the second brain's gateway
- [ ] $HOME/tools/view.py exists, executable, and `python3 -m tools.view --help` runs from inside $HOME and dispatches to the second brain's view
- [ ] $HOME/AGENTS.md gains a `## Second Brain Connection` block (auto-generated by the connect script) — content reviewed for accuracy
- [ ] All connect-script artefacts are committed atomically in $HOME for traceability (single commit, single diff to review)
- [ ] M008 smoke test (next module) passes

## Dependencies

- M006 (Pre-connect verification) — explicit
- M001 (AGENTS.md exists) — explicit, the connect script appends to it
- Second brain's `tools.setup` module must support `--connect-project` flag — this is the second brain's contract; verify before run
- Operator authorization (per epic open question — auto_connect=false → manual --connect-project run is the gate)

## Verified behavior (from tools/setup.py read 2026-05-04)

> [!success] Connect-script behavior is now known concretely
> Investigated `tools/setup.py:370-639` — the open questions below are answered with concrete behavior, not guesses.

**Collision behavior (verified):**
- `target/.mcp.json` — if exists, the `mcpServers.research-wiki` key is overwritten; other keys preserved. Idempotent re-runs land the same entry. Other servers (e.g. project-owned MCP servers) are not touched.
- `target/tools/gateway.py` + `target/tools/view.py` — preserved if they already exist WITHOUT the auto-gen marker. Re-written if they have the marker. (Marker: `# AUTO-GENERATED by second-brain setup --connect`.)
- `target/tools/__init__.py` — auto-created if missing (with marker comment); preserved otherwise.
- AGENTS.md / CLAUDE.md brain-pointer block — uses `<!-- SECOND-BRAIN-CONNECTION -->` marker. Replaces between markers if present, appends if file exists without marker, logs warn + skips if neither file exists.

**Type/group handling (CLOSED 2026-05-04):**
- ~~The connect script does NOT read the target's entry in sister-projects.yaml.~~ **Patched 2026-05-04**: `_load_sister_entry()` resolves the target path against sister-projects.yaml entries; matched entry's `type` and `group` are honored.
- ~~The injected `_BRAIN_POINTER_BLOCK` is hardcoded.~~ **Patched 2026-05-04**: a second variant `_BRAIN_POINTER_BLOCK_ROOT_OS_SETUP` is rendered when `type=root` and `group=operating-system-setup`. Generic block remains the default for all other (or unmatched) projects — backwards compatible.
- For root-modules: the OS-setup-specialized block will be injected (when $HOME/AGENTS.md or $HOME/CLAUDE.md exists). Content reframes around: OS-level IaC, second brain as methodology + verification source, SFIF stage awareness, OS-setup-relevant model + lesson queries.

**Dry-run support (CLOSED 2026-05-04):**

Verified by dry-running against $HOME (would write 4 artefacts, brain pointer skipped because $HOME/AGENTS.md doesn't exist) and against the hub itself (generic variant selected, gateway/view forwarders skipped because real ones exist, brain pointer would be added to AGENTS.md). No files modified by either dry-run.

**Forwarder error handling (NOT verified — would need runtime test):**
